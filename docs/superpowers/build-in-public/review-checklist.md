# Build-in-Public Review Checklist

Run this before any public post, devlog promotion, launch note, or community
reply that describes app status.

## Claim Accuracy

- [ ] Every "shipped" claim is in a merged build or public artifact.
- [ ] Every roadmap/prototype item is labeled as planned, in progress, or rough.
- [ ] Pricing, subscription, entitlement, and trial language matches current app/store copy.
- [ ] Dates are absolute or clearly current for the posting day.
- [ ] The post does not imply App Store availability before it exists.

## Proof

- [ ] There is a link, screenshot, clip, release note, or commit-backed devlog item.
- [ ] Screenshots/clips show the thing being claimed.
- [ ] The proof is current enough that UI/copy differences will not confuse people.
- [ ] Private files, branch names, logs, and unreleased business details are not visible.

## Privacy And Safety

- [ ] No personal library contents, addresses, coordinates, account names, or tester identities are visible.
- [ ] No signing, provisioning, certificate, analytics, or token details are visible.
- [ ] No unreleased pricing/moat/competitor notes are quoted.
- [ ] No medical, educational, accessibility, or productivity outcome is promised beyond what the app actually does.

## Community Fit

- [ ] The post follows the target community rules.
- [ ] The post is useful without requiring an install.
- [ ] There is one clear discussion question if the surface is conversational.
- [ ] There is time to reply for the next 24 hours.

## Tone

- [ ] The post sounds like a builder explaining real work, not a launch ad.
- [ ] Limitations are not hidden when they are relevant.
- [ ] The ask is small and specific.
- [ ] Cross-posted versions are adapted for the platform instead of pasted blindly.

## Final Gate

If any item is unchecked, either revise the draft or keep it private. Public
trust is worth more than filling a calendar slot.
